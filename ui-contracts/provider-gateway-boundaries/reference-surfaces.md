## Native Tool Card Rules

Native tool cards and ACP tool blocks look similar in the message list but use different data contracts.

| Question                         | Native tool card answer                                                                                         |
| -------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| Stable key                       | `toolCallId`; updates replace the existing card in place.                                                       |
| Part type                        | `tool-<toolName>`; the fallback type is `tool-unknown`.                                                         |
| Live event source                | `AgentEventPayload.data` with phase values `start`, `update`, or `result`.                                      |
| Input field                      | `args` from live start events or `arguments` from persisted `toolCall` history.                                 |
| Streaming/progress field         | `partialResult.details.progress` plus first text content fallback.                                              |
| Error field                      | `data-tool-error.error`, `details.error`, or inferred `error/status/success=false` from persisted output.       |
| Specialized renderers            | `read`, `write`, `edit`, `bash`, and `exec`; all other names render through the generic Parameters/Result card. |
| Grouping                         | Consecutive `tool-*` parts in one assistant message render inside a collapsible `ToolGroup`.                    |
| Persisted standalone tool result | `role="toolResult"` can merge into the previous assistant tool part; otherwise it remains a tool card.          |

ACP tool blocks use `data-acp-tool` with `{ toolCallId, title, status, text, tag }`. They do not expose `input`, `output`, or `progress` fields to `ChatMessages`; they render through `AcpToolBlock` and are covered in `agent-ui-contracts-via-acp.md`.
