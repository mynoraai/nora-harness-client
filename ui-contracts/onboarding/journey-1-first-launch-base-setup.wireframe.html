Provider Detail

What PM should evaluate in the required step.

required
  • 1Select primary provider or reveal more providers.
  • 2Enter key or local service address with inline format guidance.
  • 3Pick default model from known choices or verified result.
  • 4Verify connection, or intentionally skip verification.
  • 5Save provider and advance to optional customization.

Optional Steps

Lightweight setup choices that should not delay first chat.

review
Personality Tone cards, optional instructions, preview, Skip, Save & Continue.
Notifications Maybe later and Turn on notifications both lead to Completion.
Completion Summarizes provider/model and sends user into Chat.

Engineering Readiness

Current behavior exists; automated proof is uneven.

coverage gap
Guarded now Welcome paints, Get started opens Provider, default provider selected, Continue disabled, Back returns.
Needs stronger guard Full provider save, personality skip/save, notification skip/allow, final Chat handoff.
PM impact The product flow exists, but release confidence should account for unguarded recovery states.

State Language

Recommended PM vocabulary for the setup path.

proposed copy
Welcome -> Provider idle -> Verifying -> Verified -> Ready for Chat

Alternate provider path: valid input -> skip verification -> provider saved -> optional steps.

PM Decisions

Choices this artifact is meant to unblock.

PM review
Blocking: Is Provider setup the only required step before Chat, or can users enter Chat without it?
Decision: Should Skip verification stay visible, move behind help, or be limited to local/network-constrained cases?
Decision: Should Completion always go to Chat, or offer Chat / Code entry?

Discussion Boundaries

Keep PM review focused.

scope
Discuss Required versus optional steps, banner visibility, skip-verification copy, Chat handoff.
Do not discuss Main app IA, Voice, Channels, Advanced settings, provider API internals.
Fixed Notifications are not a hard blocker; users can customize later.

Provider Failure Recovery

Failures should keep the user in context.

important
Key rejected Tell user to check copied key; keep fields editable.
Network failure Ask user to check connection and retry; show skip only when input is valid.
Local unreachable Ask user to confirm the local service address and retry.

Optional Step Recovery

Optional steps should never trap the user.

recovery
Personality save fails Show retry and keep Skip available.
Notification denied Continue to Completion and explain it can be changed later.
Local banner visible Do not block setup unless the required Provider save cannot finish.

Definition Of Success

What "ready" should mean for Journey 1.

success
  • 1At least one provider/model is saved or intentionally accepted as unverified.
  • 2Personality and notification choices do not block first chat.
  • 3Completion leads to a usable Chat draft instead of an empty app state.
  • 4Known coverage gaps are tracked before release-critical signoff.
