Journey 02 - Main App Startup, Gateway Status, Update Prompt

Returning users should understand app readiness, connection health, saved mode, and update availability without being blocked from work.

Status: needs verificationVerification point: Verify the startup priority order for health, updates, and returning users.

Jump list

Main path

4 product wireframe frames

Launch shell

App opens directly to the last useful mode with sessions visible while checks start.

  • No marketing or splash page.
  • Recent sessions appear before checks finish.

Health banner

A compact health banner reports starting, ready, degraded, or offline without blocking navigation.

  • Health is above update prompts.
  • Retry and diagnostics are visible.

Mode restore

Saved mode and selected workspace return with clear fallbacks when context is missing.

  • Chat or Code state is explicit.
  • Missing workspace is recoverable.

Update prompt

Update availability appears as a small prompt with release notes, later, and restart actions.

  • Once per version.
  • Does not hijack active work.

Detail states

3 product wireframe frames

Startup priority

Critical work-blocking health problems outrank optional update prompts.

  • Health first, update second.
  • Critical states get the primary action.

Returning sessions

Recent conversations and code sessions are visible while background checks run.

  • Skeleton rows avoid blank sidebar.
  • Content becomes interactive as it loads.

Offline work

When local service is unavailable, local navigation remains usable and disabled actions explain why.

  • Read-only work stays available.
  • Send is disabled with a reason.

Error recovery

2 product wireframe frames

Update failed

The update prompt keeps the current version usable and offers retry plus release notes.

  • No forced loop.
  • Current app remains usable.

Service degraded

Users can retry, open diagnostics, or continue with limited features.

  • Severity label is clear.
  • Limited mode is explicit.

Journey notes

User outcome, constraints, evidence gaps
User outcome

Startup feels dependable even when health checks are still running. Update messaging is visible without hijacking the session.

Constraints

More visible health states require careful copy. Too many startup prompts can compete for attention.

Evidence gaps

Should update prompts be hidden during active code sessions until the session is idle?
