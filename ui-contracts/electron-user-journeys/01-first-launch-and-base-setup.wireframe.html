Journey 01 - First Launch And Base Setup

New users land in a guided setup that gets a local workspace, provider access, preferences, and notification readiness in place before the first conversation.

Maturity: partialPM decision: Confirm the minimum first-run setup that must happen before the user reaches chat.

Jump list

Main path

5 wireframe frames

Welcome

First launch starts inside the product shell, with guided setup as the primary action.

  • Skip is available but names consequences.
  • No marketing hero or separate landing page.

Base folder

User confirms where local workspace and agent files can be created safely.

  • Default path visible and changeable.
  • Access test controls continue state.

Provider access

Provider setup can be completed now or deferred with a visible first-chat warning.

  • Private credentials, testable readiness.
  • Skip leaves Send disabled later.

Identity and tone

Keep profile lightweight: name, tone, proactive suggestions, notification timing.

  • Short defaults, not a long profile form.
  • Notification permission is contextual.

Ready screen

Finish lands in Chat with a setup summary and any remaining next action.

  • Composer readiness mirrors model state.
  • Setup can be reopened later.

Detail states

3 wireframe frames

Skip behavior

Skipping setup opens Chat but keeps model/setup debt visible and actionable.

  • Missing provider card stays near composer.
  • User can resume setup from Chat.

Notification prompt

Permission is introduced only when reminders or scheduled work makes it meaningful.

  • Ask later is a valid state.
  • System prompt is not first-launch noise.

Reopen setup

Settings exposes the same setup choices after onboarding.

  • Users can change folder/provider/tone.
  • Incomplete setup remains resumable.

Recovery states

3 wireframe frames

Folder access failed

The blocking failure offers retry, choose another folder, or inspect permission guidance.

  • Continue disabled until safe write access exists.

Provider test failed

User keeps their draft key state and can retry, edit, or continue without model.

  • Error is secret-safe and field-local.

First chat gated

If setup was skipped, the composer explains exactly what must be fixed before Send.

  • Forbidden action appears disabled, not missing.

PM decisions

3 decision frames

Mandatory setup

Decide whether provider configuration is required before first chat or only recommended.

  • Impacts activation speed and first-message success.

Profile depth

Decide whether identity/tone belongs in setup or after first successful conversation.

  • Impacts perceived setup length.

Notification timing

Confirm whether notification permission waits for scheduled tasks or appears in setup.

  • Blocking because OS prompt timing affects trust.

Review notes

Benefits, tradeoffs, open questions
Benefits

PM can inspect the actual setup screen structure while keeping the map-like canvas overview.

Tradeoffs

Wireframe cards are denser than text cards, so each card focuses on the state PM must judge.

Open questions

Should provider setup block first chat, and should notification permission wait until scheduled tasks?
