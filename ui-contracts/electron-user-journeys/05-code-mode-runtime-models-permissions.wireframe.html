Journey 05 - Code Mode Runtime, Models, Permissions

Code sessions need clear runtime choice, model readiness, tool permission prompts, and visible health during long-running work.

Maturity: partialPM decision: Confirm how much runtime and permission state should be visible before a code task starts.

Jump list

Main path

4 frames

Ready check

Composer shows selected runtime, selected model, and whether tools can be used safely.

  • Ready badge
  • Missing item callout

Send code task

User submits a task and the session locks to the chosen runtime for consistent execution.

  • Runtime label persists
  • Task starts inline

Permission request

Sensitive actions pause with an approval card explaining action, scope, and options.

  • Approve once
  • Deny
  • Change policy

Progress events

The transcript shows thinking, file changes, terminal work, and completion in a scan-friendly sequence.

  • Grouped events
  • Stop available

Detail states

3 frames

Model missing

User gets a direct path to choose a model or open provider settings.

  • No hidden failure

Runtime unavailable

The runtime selector explains unavailable choices and offers retry or fallback.

  • Reason visible

Policy summary

Session header shows whether approvals are strict, ask-first, or more permissive.

  • Safety always visible

Error recovery

2 frames

Denied action

The session continues with an explanation and an option to adjust the request.

  • No lost context

Runtime crash

The app keeps transcript state and offers restart or choose another runtime.

  • Task not silently abandoned

Review notes

Benefits, tradeoffs, open questions
Benefits

Users see safety tradeoffs before work starts. Permission pauses are explainable instead of surprising.

Tradeoffs

More health badges can make the composer dense. Permission language must be short but precise.

Open questions

Should permissive policies require an extra confirmation for new workspaces?
