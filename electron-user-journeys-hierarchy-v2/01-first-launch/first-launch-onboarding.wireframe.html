Journey 01: First Launch Onboarding

User journey wireframe for the implemented first-run path from fresh install to first usable Chat. Product frames describe the implemented first-run path and current app behavior.

Maturity: implemented Journey: 01 first launch Record: required steps, skip paths, final handoff

Jump list

Main happy path

Fresh install to first usable Chat

1 Welcome

User lands in a centered onboarding shell. One primary action starts setup.

  • Step progress shows Step 1 of 4.
  • Get started is the only action on the first screen.

2 Provider Setup

User chooses a provider, enters a key or local endpoint, chooses a model, then verifies or explicitly skips verification.

  • Continue is gated until verification is satisfied or skipped.
  • Back returns to Welcome and does not save the verified provider.

3 Notifications

User chooses whether to ask for system notifications now or continue without them.

  • Maybe later advances immediately.
  • Turn on notifications requests OS permission, then advances.

4 Completion

User sees provider/model confirmation and clicks Start chatting.

  • Completion confirms the provider saved in step 2.
  • The onboarding-complete flag is written only after Start chatting.

5 First Chat Surface

The app opens the normal Chat surface with a draft conversation ready.

  • This is the end of Journey 01.
  • Code workspace creation starts a separate journey.

Branch journeys

Full product screens for paths that branch from the main journey
1 Welcome

Branch context starts from the same fresh-install entry.

  • User clicks Get started.
  • The path moves to Provider Setup.
2 Provider Setup

The branch point is the provider form, before verification has completed.

  • User enters a valid-looking key.
  • Verify key starts the success or failure branch.

2.1 Verification success

After the user verifies a valid provider key, Provider Setup stays in place and changes into a verified state.

  • The key remains masked and shows Verified.
  • Verify again and Continue are both available.
  • The user can open the model selector before continuing.

2.1.1 Verified model selection

The verified provider path can continue into the model dropdown before saving Provider Setup.

  • The dropdown expands in the same Provider Setup step.
  • The selected model has a checkmark.
  • Continue saves the selected provider and model.

2.2 Missing credential

The user has not entered a provider key yet. Verification and Continue remain blocked inside Provider Setup.

  • The field stays editable.
  • The inline error explains that a key is required before verification.

2.3 Remote key validation failed

The user typed a remote provider key that does not pass local shape validation.

  • The key remains visible as typed.
  • The user must edit the key before verification can proceed.

2.4 Local endpoint validation failed

The user chose Local Models but entered an endpoint without a valid URL scheme.

  • The URL field uses local endpoint language.
  • Test connection is blocked until the URL is corrected.

2.5 Key rejected

Verification ran, but the provider rejected the key. The user remains in Provider Setup with retry and skip available.

  • The credential is preserved.
  • The recovery path is edit, retry, switch provider, or use advanced skip.

2.6 Provider unreachable

Verification cannot reach the provider. The same Provider Setup screen stays open with a network recovery message.

  • The user can retry after connection changes.
  • Advanced skip remains available when the input is otherwise valid.

2.7 Unexpected validation result

The provider result cannot be mapped to a more specific rejection or network message.

  • The message stays inline with the credential form.
  • The user can retry, edit, switch provider, or skip verification.

2.8 Skip confirmation

The advanced skip action opens a confirmation over Provider Setup before Continue is unlocked.

  • Cancel returns to the unverified Provider Setup screen.
  • Confirming skip changes the form state to Verification skipped.

2.8.1 Verification skipped

The user confirmed the advanced skip branch. Provider Setup now permits Continue without a successful verification result.

  • The status line shows Verification skipped.
  • The first Chat request may surface the provider problem later.

2.9 Provider save failed

Continue attempted to save provider settings but failed. The user stays in Provider Setup with the same values.

  • The verified state is preserved.
  • The user can retry Continue or edit before retrying.

3.1 Notification permission unresolved

Notification permission can be blocked or unresolved, but onboarding keeps moving toward completion.

  • The copy points users to System Settings for later changes.
  • This branch does not block first chat.

4.1 Start chatting handoff

The completion screen is the final onboarding frame before the app opens Chat.

  • Start chatting writes the completion state.
  • A draft Chat session is prepared if needed.

4.2 Later launch after completion

After completion is stored, later app launches skip onboarding and open the main Chat surface.

  • Welcome does not appear again during normal launch.
  • Resetting onboarding is a separate debug recovery action.

4.3 Reset onboarding

If onboarding is reset, the next launch returns to the Welcome screen instead of the Chat surface.

  • The user re-enters Journey 1 from the beginning.
  • Provider setup must be completed again before first Chat.

Detail states

Implemented branches inside onboarding

2.10 Provider entry and edits

Provider setup starts with Anthropic selected, an empty key, and the first default model selected. Switching provider clears the key, returned models, verification, skip state, and inline error.

idle Empty key

Verify key and Continue are disabled. Helper text says the user can change this later in Settings.

invalid Typed but blocked

Malformed remote keys and bad local URLs show inline validation before any verification call can run.

ready Inputs complete

Verify key, or Test connection for Local Models, becomes enabled. Continue is still disabled.

2.11 More Models and provider switching

More Models expands the provider grid. OpenRouter adds a nested provider dropdown; Local Models changes the credential field into a service URL and the model field into free text.

more Expanded grid

Additional provider cards appear in the same step; no navigation occurs.

openrouter Provider dropdown

The form adds a provider select above the API key and keeps model selection in the same card.

local Local endpoint

The API key input becomes a URL field and the verify action becomes Test connection.

2.12 Verification route

The verification action runs inside the same form. Remote providers call the verifier; Local Models only probes transport reachability.

verifying Checking...

The credential and model fields are disabled. Remote button copy is Checking...; local copy is Testing....

verified Success

The credential row shows Verified. Continue becomes enabled, Verify again/Test again remains available, and the model selector can be opened.

error Failure

The form keeps the typed value and shows an inline recovery message.

2.13 Skip verification route

Skip verification appears only after inputs are otherwise valid and verification has not been satisfied. The browser-style confirm dialog is the branch point.

confirm User clicks skip

The confirmation warns that first-message errors may appear later if the key is wrong or the network blocks verification.

cancel User cancels

The form returns to the unverified state. Continue remains disabled.

skipped User confirms

The row shows Verification skipped. Continue becomes enabled without calling the verifier.

2.14 Provider save, back, and advance

Back returns to Welcome without persisting the provider. Continue saves the provider only after verification is satisfied or skipped, then advances to Notifications.

back Back before Continue

Welcome reappears and no provider is saved.

submitting Saving and continuing...

The Continue button changes copy while provider config is saved.

advance Save succeeds

The app records connected provider/model and moves to Step 3 of 4.

3.2 Notification branches

Maybe later advances immediately. Turn on notifications asks the desktop app to request permission; the step advances after the request completes.

skip Maybe later

No OS permission request is made, and Completion appears.

request Turn on notifications

The app requests notification permission and may show a test notification.

continue Request resolves

The flow advances regardless of the returned permission status.

2.15 Provider action language

Checking...Remote verification is running and form inputs are disabled.

2.16 Provider status language

Status copy appears beside the field or under the form so the user sees what changed without leaving Provider setup.

VerifiedThe key or local endpoint has passed verification and Continue is enabled.
Default model menuAfter verification, the selected model is shown in a dropdown and can be changed before Continue.
Verification skippedThe user confirmed the advanced skip branch and may continue without verification.
Settings helperNeutral helper copy says the user can change provider settings later.
Skip warningThe confirmation warns that the first message may fail if the key is wrong.

0.1 Whole-flow language

  • Notifications uses Turn on notifications and Maybe later.
  • Completion uses Start chatting.
