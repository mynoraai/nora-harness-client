Journey 01: First Launch Onboarding

User journey wireframe for the implemented first-run path from fresh install to first usable Chat. Product frames describe the implemented first-run path and current app behavior.

Maturity: implemented Journey: 01 first launch Record: required steps, skip paths, final handoff

Jump list

Main happy path

Fresh install to first usable Chat

1. Welcome

User lands in a centered onboarding shell. One primary action starts setup.

  • Step progress is visible before any app shell appears.
  • No alternate action competes with Get started.

2. Provider Setup

User picks an AI service, enters a key or endpoint, chooses a default model, and verifies.

  • Continue remains disabled until verification is satisfied.
  • Skip verification exists, but is framed as advanced.

3. Personality

User chooses tone, optionally adds instructions, previews the initial identity, then saves or skips.

  • Saving creates the initial agent identity.
  • Skip lets setup continue without blocking value.

4. Notifications

User chooses whether to ask for system notifications now or continue without them.

  • Both actions advance the flow.
  • The operating system prompt is outside this product frame.

5. Completion

User sees provider/model confirmation and clicks Start chatting.

  • Completion is the moment the first-run gate is satisfied.
  • Provider/model summary reassures the user before handoff.

6. First Chat Surface

The app opens a new Chat conversation with composer ready.

  • This is the end of Journey 01.
  • Code workspace creation starts a separate journey.

Detail states

Onboarding state changes that are not separate journeys

1. Provider setup pending

The user has entered Provider setup. The UI preserves the selected provider, model, API key, and base URL inputs. Continue remains controlled before verification; the user can verify, change provider, or configure later.

  • The form shows required, verifying, and verified states inline.
  • Skipping does not block onboarding completion, but the next conversation can enter the provider-readiness repair path.

2. Personality and local preference saved

After the user completes personality, notifications, and local-run preferences, step progress advances and the UI shows lightweight saving-to-saved feedback.

  • Optional fields do not force the user to stay.
  • Going back preserves entered content.

3. First usable Chat handoff

After the user clicks Finish, the onboarding shell closes and the normal Chat home opens. The client records onboarding complete and shows the main surface ready for task input.

  • The sidebar, composer, and runtime/model chips appear together.
  • This is the end of Journey 01; later workspace creation belongs to a separate journey.

State language

User-facing copy for each onboarding state

1. Primary actions

Buttons such as Get started, Continue, and Finish setup express only the current step advance and do not carry extra explanation.

  • Progress copy such as Step 1 of 5 stays fixed at the top.
  • Verify connection is used for provider verification and is not conflated with Save.

2. Optional actions

Copy such as Skip for now, Maybe later, and Back clearly indicates that the current context will not be lost.

  • When the user skips provider or notification setup, the UI gives lightweight confirmation.
  • Back actions remain lower priority.

3. Status copy

States such as Saving, Verified, Complete, and Needs setup directly describe the current client state.

  • State appears near the relevant form or step first.
  • Do not use decision-process wording.

Errors and recovery

Recoverable onboarding branches

1. Provider validation failed

If API key, base URL, or model verification fails, inputs are preserved and the form shows an inline error. The user can verify again, change provider, or configure later.

  • Failure does not clear fields.
  • After skipping, the user returns to the main app; starting a later task can trigger the repair entry again.

2. Notification permission declined

If the user denies or closes system notification permission, onboarding still completes. The setting remains disabled and can be enabled later in Settings.

  • A system denial is not treated as flow failure.
  • Explanatory entry points stay near the notification step.

3. Onboarding resume

After an unexpected quit or refresh, the client restores the unfinished step or re-enters Welcome. Saved local configuration is not requested again.

  • Recoverable state takes priority over starting from scratch.
  • After completion, later launches no longer show onboarding.
